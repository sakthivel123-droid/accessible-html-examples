WCAG 2.2 – 2.5.3 Label in Name

🚫 Bad Examples – Mismatch Between Label and Accessible Name

Issue: Visible label says “Search”, but screen reader says “Find”. A voice user saying “Click Search” would fail because the accessible name doesn’t match.

Issue: “Submit” is visible, but the accessible name “Send Form” replaces it. Voice command “Click Submit” will not work.

Issue: Placeholder text “Enter name” differs from accessible label “Type your full name”. Users saying “Enter name” won’t activate this field.

✅ Good Examples – Label Text Matches Accessible Name

Explanation: The visible label “Search” matches the accessible name “Search”. Voice command “Click Search” and screen readers both read correctly.

Explanation: “Search this site” includes visible word “Search”. This still passes, as the visible text is part of the accessible name.

Explanation: The label “Username” is both visible and used as accessible name. Perfect match for screen readers and voice input.

Explanation: Even though the icon is visual only, the accessible name “Close” correctly reflects the intended visible function.

📘 Summary of All Scenarios

  • Requirement: The visible text label must be included in the accessible name.
  • Bad Scenarios:
    • Visible text and accessible name don’t match (e.g., “Search” vs. “Find”).
    • aria-label replaces visible text with different words.
    • Placeholder text differs from label or accessible name.
  • Good Scenarios:
    • Accessible name matches visible label exactly.
    • Accessible name includes visible label (e.g., “Search this site”).
    • Icon buttons have aria-labels that describe visible purpose (e.g., “Close”).
    • Form inputs use visible labels that match accessible names.
